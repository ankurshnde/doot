§02 · FOUR RINGS OF INNOVATION

Coordination, not control.

The Kumbh governance challenge sits in concentric rings. Infrastructure and crowd safety remain with existing protocols. The agentic layer solves the middle rings — access and experience — where the work is coordination, consent and multi-stakeholder orchestration. Click a ring to explore.

§03 · THREE CORE PRINCIPLES

One pilgrim,
one agent.

i
PILGRIM LAYER

Kumbh Doot

A personal, voice-first AI envoy. Runs locally on phone, feature phone, or voice-only via IVR. Multilingual across 20+ Indian languages, low-literacy tolerant, offline-capable. Encrypted personal data stays on-device; a secure cloud replica handles complex reasoning and recovery.

ii
CIVIC & COMMERCIAL LAYER

Coordinated
counterparties

Government departments, transport, health, police, camps and local vendors operate coordinated agents connected to existing databases via MCP. They negotiate on behalf of principals — no heavy LLM cost, no central command-and-control.

iii
SOVEREIGN PLATFORM

India Stack 2.0

Extends DPI from transactional (Aadhaar, UPI, DigiLocker, ABDM) to an agentic trust layer. Consent-driven, DEPA-compliant. No central citizen database. Identity verified, never pooled.

§05 · TRUST INFRASTRUCTURE

Delegated authority,
enforced by code.

The agent holds no intrinsic power. All authority flows from the citizen, is constrained by law, and enforced by technology. Seven pillars define the boundary.

01 / IDENTITY

Identity-bound
Delegation

Each agent is cryptographically bound to the citizen's digital identity. Every action presents a verifiable claim: Agent A, for Citizen C, for Purpose P, under Scope S.

02 / CONSENT

Least
Privilege

Fine-grained, revocable consent. Permissions are purpose-specific, time-limited and capability-scoped. No residual access after revocation.

03 / TOOLS

Secure Tool
Execution

The model proposes; only policy-compliant tools execute. All network calls, forms, payments and retrievals pass through a validated tool layer. External input is hostile by default.

04 / HITL

Human in the
Loop

Financial transfers, legally binding documents, identity changes and third-party sharing require explicit confirmation — biometric, PIN or crypto signature — before authorization.

05 / POLICY

Policy-bound
Reasoning

A machine-readable policy layer encodes law, financial limits and data-protection rules. Even if the model proposes an unsafe plan, the engine prevents execution.

06 / PRIVACY

On-device &
TEE

Agents run on-device or inside Trusted Execution Environments. Memory, inputs and reasoning state are encrypted in use — inaccessible even to infrastructure operators. Verifiable via remote attestation.

07 / AUDIT

Tamper-evident
Forensics

Every data access, tool invocation and transaction is written to tamper-evident logs — visible to the citizen, accessible to regulators where legally required. Dispute resolution has evidence by default.

∴ / DEFAULTS

Defaults as
Governance

At population scale, factory settings are the policy. Privacy-by-design ships with no data-sharing default-on. The vulnerable are protected by architecture, not by their ability to configure.

§06 · DUAL AGENT FRAMEWORK

Edge-first,
cloud when necessary.

Two synchronised agents. Sensitive data stays local. The cloud handles what the edge cannot. At population scale, edge-first is the only sustainable path — intelligence scales with devices already in citizens' hands.

PRIMARY · ON-DEVICE · NPU

Phone Agent

  • Local LLM inference on-device chips
  • Personal data never leaves the phone
  • Full offline capability for core flows
  • Voice interaction & conversation history
  • Lightweight on-device personalisation
  • Degrades gracefully without connectivity
E2E ENCRYPTED SYNC
OPTIONAL · TEE · SOVEREIGN CLOUD

Cloud Replica

  • Complex reasoning & multi-agent coordination
  • Runs inside Trusted Execution Environments
  • Remote attestation — verify approved code
  • Cross-device continuity via DigiLocker
  • Multi-provider model — sovereign + private
  • Falls back to edge-only on attestation failure

§08 · RISKS & MITIGATIONS

Honest about
what can break.

Technical

Connectivity at scale
Mobile networks may not support 10M concurrent users on peak snan days.
Telecom pre-coordination for temporary towers; IVR/USSD low-bandwidth fallbacks; edge voice models reduce round-trips.
Low-resource languages
Voice accuracy in Bhojpuri, Maithili or other dialects lags Hindi/English.
Dialect-specific fine-tuning on Bhashini datasets; human fallback operators; per-language error monitoring during pilot.

Adoption

Pilgrim trust & uptake
Many pilgrims may be skeptical of AI agents or unwilling to register.
Registration entirely voluntary; unregistered pilgrims retain full access; community leaders and state-camp volunteers as onboarding ambassadors.
Vendor participation
Small vendors may lack capability or willingness to integrate.
Lightweight integration via existing UPI; voice-based vendor registration; incentive programs for early adopters.

Governance

Agent liability
If an agent provides incorrect medical guidance or routes a pilgrim into danger, who is responsible?
Agents provide recommendations not directives; medical/emergency routing validated against curated gov sources, not open LLM output; full audit trails.
Political & cultural
Risk of appearing to gatekeep a traditionally open religious gathering.
Kumbh Pass framed as optional enabler, not requirement; terminology avoids "visa" framing; unregistered pilgrims retain full physical access.
